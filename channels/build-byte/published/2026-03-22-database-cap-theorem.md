## Пост (копировать в Telegram)

**CAP и базы данных** — коротко о теореме и зачем она нужна при выборе и проектировании хранилища.

**Архитектура**

CAP описывает компромиссы в **распределённых** системах: нельзя одновременно получить все три свойства сразу — приходится осознанно выбирать, на чём строить систему.

**Что такое CAP**

- **C (Consistency)** — согласованность: после записи чтение возвращает актуальные данные
- **A (Availability)** — доступность: запросы получают ответ. Данные доступны всегда.
- **P (Partition tolerance)** — устойчивость к **разделению**: репликация данных на несколько узлов.

**Как это работает**

Какие бывают сочетания 
- capability + availability - как правило один узер базы данных. Данный в ней консистентны и доступны. Реплики отсутствуют.
- availability + partition -  в этот варианте мы имеет кластер базы данных, который может быть доступен, но не всегда консистентен т.к присутствует задержка репликации и в этот момент данные могут быть не консистентны.
- capability + partition - данный вариает встречается редко. Данный реплицируются. Данные консистентны, но доступны не всегда. Одним из примеров является Apache Kafka. Сообщение не будет доступно потребителю пока не будет реплицировано на все узлы.

**Зачем это нужно**

- **выбор БД** — понимать какие сочетания свойств нам нужны и какие варианты доступны.
- **ожидания продукта** — заранее договориться, допустимы ли задержки репликации и «устаревшие» чтения;
- **дизайн отказоустойчивости** — грамотно спроектировать систему так, чтобы она могла работать в условиях сбоев и репликации.
